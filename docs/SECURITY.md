# Безопасность

## Модель авторизации

- **Пользователи.** Каждый запрос к `/api/*` обязан содержать заголовок
  `x-telegram-init-data`. Подпись проверяется HMAC-SHA256 по схеме Telegram
  WebApp (`auth.js`). Просроченные (> 24 ч) и поддельные подписи отклоняются.
- **Администратор.** Защищённые маршруты (`/api/admin/*`) дополнительно
  проверяют, что `telegramUser.id === ADMIN_ID`.

## Защита от абьюза

- **Rate-limiting.** Общий лимит на `/api` по IP и отдельный, более строгий,
  лимит на дорогие ИИ-запросы — по Telegram-пользователю (`middleware/security.js`).
  Настраивается через переменные `RATE_LIMIT_*`.
- **Security-заголовки.** `X-Content-Type-Options`, `Referrer-Policy`,
  `Permissions-Policy` и др. Frame-политики намеренно НЕ ограничиваются —
  иначе сломается отображение Mini App во встроенном webview Telegram.
- **Лимит размера запроса.** JSON ограничен 1 МБ, загрузка файлов — 10 МБ.

## Секреты

- Все секреты — только в `.env` (в git не попадает, см. `.gitignore`).
- Кэш токена GigaChat (`.giga_token.json`) содержит секрет и исключён из git.
- Загруженные чеки (`uploads/`) содержат персональные данные и исключены из git.

## Известные ограничения и рекомендации

1. **TLS для GigaChat.** Сейчас в `index.js` глобально отключена проверка TLS
   (`NODE_TLS_REJECT_UNAUTHORIZED=0`), т.к. GigaChat использует сертификаты
   «Russian Trusted CA». **Рекомендация:** установить корневые сертификаты
   Минцифры и задать `NODE_EXTRA_CA_CERTS`, после чего убрать глобальное отключение.
2. **Публичный доступ к чекам.** Каталог `uploads/` отдаётся статикой и доступен
   по прямой ссылке (имена файлов случайны). Для повышенной защиты разместите
   `uploads/` за авторизацией reverse-proxy или храните чеки в приватном хранилище.
3. **Один инстанс для rate-limit.** Лимитер хранит счётчики в памяти. При
   горизонтальном масштабировании используйте общий бэкенд (Redis).
